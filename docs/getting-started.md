# Getting started (framework-agnostic)

Use this guide when you want to integrate Goose into an existing Python app without committing to a framework-specific
adapter first.

This is the shortest path to a first passing Goose test:

1. run `goose init`
2. point Goose at `query(...) -> AgentResponse`
3. register a Goose fixture
4. write one `goose.case(...)`
5. branch out into tools, live chat, and hot reload

If you already have a LangChain or LangGraph agent, keep this page as the mental model and use
[`integrations/langchain.md`](integrations/langchain.md) for the adapter-specific version.

## 1. Install

Required for the first test run:

```bash
pip install llm-goose
```

Optional for the browser UI:

```bash
npm install -g @llm-goose/dashboard-cli
```

## 2. Run `goose init`

```bash
goose init
```

That creates the fixed `gooseapp/` layout Goose expects:

```text
gooseapp/
├── README.md
├── __init__.py
├── app.py
├── conftest.py
└── tests/
    ├── __init__.py
    └── test_example.py
```

What each file is for:

- `gooseapp/app.py` - tools, chat agents, and hot reload settings
- `gooseapp/conftest.py` - test fixtures and the link to your real `query(...)` function
- `gooseapp/tests/` - `test_*.py` modules with `goose.case(...)` cases
- `gooseapp/README.md` - local quick reference generated by the scaffold

For scaffold details, overwrite behavior, and validation rules, see [`goose-init.md`](goose-init.md).

## 3. Create `query(...) -> AgentResponse`

Your Goose fixture does not talk to the agent directly. It calls a `query(...)` function in your app code. Goose sends
the exact `query=` string from `goose.case(...)` into that function and expects an
`AgentResponse` back.

Minimal contract:

```python
from goose.testing.models.messages import AgentResponse, Message, ToolCall


def query(message: str) -> AgentResponse:
    result = run_my_agent(message)

    return AgentResponse(
        messages=[
            Message(
                type="ai",
                content=result.text,
                tool_calls=[
                    ToolCall(
                        name="search_products",
                        args={"query": message},
                    )
                ],
            )
        ]
    )
```

What Goose requires:

- the function receives the exact user message as `message: str`
- it returns `goose.testing.models.messages.AgentResponse`
- you should include at least one final `Message(type="ai", content="...")`
- include `tool_calls=[...]` when you want `expected_tool_calls=[...]` to be validated

If you already have a LangChain response payload, do not rebuild it by hand. Use
`AgentResponse.from_langchain(...)` instead and follow [`integrations/langchain.md`](integrations/langchain.md).

## 4. Register the Goose fixture in `gooseapp/conftest.py`

The simplest pattern is to name the fixture `goose` directly:

```python
from goose.testing import Goose, fixture
from my_agent import query


@fixture()
def goose() -> Goose:
    return Goose(
        agent_query_func=query,
        validator_model="gpt-4o-mini",
    )
```

If you prefer a different Python function name, override the registered fixture name explicitly:

```python
@fixture(name="goose")
def goose_fixture() -> Goose:
    return Goose(
        agent_query_func=query,
        validator_model="gpt-4o-mini",
    )
```

### How fixture injection actually works

This is the exact matching rule from `goose/testing/fixtures.py`:

- Goose injects fixtures by **test parameter name**
- `@fixture(name="...")` sets that injection name explicitly
- if `name=` is omitted, Goose uses the fixture function name
- the type annotation (`Goose`) is recommended, but matching does **not** happen by type

So this:

```python
@fixture(name="goose")
def goose_fixture() -> Goose:
    ...


def test_price_lookup(goose: Goose) -> None:
    ...
```

works because the test parameter is named `goose`, not because the Python function is named `goose_fixture`.

## 5. You can register more than one Goose fixture

This is useful when you want to test different agents, environments, or hook setups:

```python
@fixture(name="sales_goose")
def sales_goose_fixture() -> Goose:
    return Goose(agent_query_func=query_sales_agent)


@fixture(name="staging_goose")
def staging_goose_fixture() -> Goose:
    return Goose(agent_query_func=query_staging_agent)
```

Then your tests opt into the right fixture by parameter name:

```python
from goose.testing import Goose


def test_sales_quote_flow(sales_goose: Goose) -> None:
    sales_goose.case(
        query="Prepare a quote for hiking boots",
        expectations=["Agent prepares a quote for hiking boots"],
    )
```

In practice, keep one Goose fixture parameter per test so it is always obvious which agent or environment the case
targets.

## 6. Write the first Goose test

```python
from goose.testing import Goose
from my_agent.tools import get_product_details


def test_price_lookup(goose: Goose) -> None:
    goose.case(
        query="What's the price of Hiking Boots?",
        expectations=[
            "Agent provides the price for Hiking Boots",
            "Response is direct and factual",
        ],
        expected_tool_calls=[get_product_details],
    )
```

What is happening here:

- `goose` is the injected fixture from `gooseapp/conftest.py`
- `query=` becomes the exact `message` passed to your real `query(...)` function
- `expectations=` are validated in natural language
- `expected_tool_calls=` checks that the named tools were actually called

## 7. Run the first loop

```bash
goose test list gooseapp.tests
goose test run -v gooseapp.tests
```

Start there. You do **not** need to customize `gooseapp/app.py` before your first passing test because `goose init`
already generated a valid placeholder `GooseApp`.

Once you want the full runtime loop:

```bash
goose api
goose-dashboard
```

- `goose api` starts the Python backend on `127.0.0.1:8730`
- `goose-dashboard` starts the UI on `http://localhost:8729`

See [`running-goose.md`](running-goose.md) for ports, runtime behavior, and reload details.

## 8. What to do in `gooseapp/app.py` next

After the first passing test, `gooseapp/app.py` becomes the place where you expose more of your app to Goose.

### Configure tools for the Tooling view

Register tools directly:

```python
from goose import GooseApp
from my_agent.tools import get_product_details, search_products


app = GooseApp(
    tools=[search_products, get_product_details],
)
```

Those imports come from **your app code**, not from Goose. `GooseApp.tools` expects a list of tool callables
(`Sequence[Callable[..., Any]]`), so there is no Goose-specific `Tool` base class to inherit from.

The best-supported shape is a LangChain-style tool object or `@tool`-decorated function because Goose can read:

- `name`
- `description`
- optional `args_schema`
- `invoke(...)` / `ainvoke(...)` when available

In practice, the easiest way to define them is in your own module, for example `my_agent/tools.py`:

```python
from langchain_core.tools import tool


@tool
def search_products(query: str) -> str:
    """Search the product catalog by free-text query."""
    ...


@tool
def get_product_details(product_id: str) -> str:
    """Return detailed information for one product."""
    ...
```

Then import those tools into `gooseapp/app.py`:

```python
from my_agent.tools import get_product_details, search_products
```

Plain Python callables also work, but the Tooling view gets the richest metadata when the tool exposes LangChain-style
attributes such as `name`, `description`, and `args_schema`.

Or group them in the UI:

```python
app = GooseApp(
    tool_groups={
        "Catalog": [search_products, get_product_details],
    },
)
```

### Configure in-app chat agents

Add agents when you want live chat in the dashboard:

```python
from my_agent.chat import support_agent


app = GooseApp(
    agents=[support_agent],
)
```

`support_agent` also comes from **your app code**. `GooseApp.agents` accepts agent objects. Goose validates a unique
`name` up front, then the live chat runtime expects one of these protocols:

- Goose-native agents with `astream_goose(...)`
- LangChain-style agents exposing `astream({"messages": messages}, stream_mode="messages")`

If you are on the framework-agnostic path, the clearest contract is the Goose-native protocol from
`goose.chatting.agent_protocol`:

```python
from collections.abc import AsyncIterator

from goose.chatting.agent_protocol import GooseAgentEvent, GooseChatAgent
from goose.chatting.api.schema import Conversation
from goose.testing.models.messages import Message


class SupportAgent:
    name = "Support Agent"

    async def astream_goose(
        self,
        *,
        conversation: Conversation,
        messages: list[Message],
    ) -> AsyncIterator[GooseAgentEvent]:
        latest = messages[-1].content if messages else ""
        yield GooseAgentEvent(type="token", data={"content": f"Support heard: {latest}"})
        yield GooseAgentEvent(type="message_end")


support_agent: GooseChatAgent = SupportAgent()
```

That is the Goose-native protocol in plain words:

- `name: str`
- `astream_goose(*, conversation, messages) -> AsyncIterator[GooseAgentEvent]`

If you already have a LangChain or LangGraph agent exposing
`astream({"messages": messages}, stream_mode="messages")`, you can register that object in `agents=[...]` too. Goose
currently consumes `AIMessageChunk` and `ToolMessage` items from that stream. The LangChain-specific shape is covered
in [`integrations/langchain.md`](integrations/langchain.md).

### Configure hot reload

Use `reload_targets` for the packages you edit often and `reload_exclude` for modules that should stay stable:

```python
app = GooseApp(
    reload_targets=["my_agent", "shared_utils"],
    reload_exclude=["my_agent.models", "my_agent.db"],
)
```

Goose always includes `gooseapp` automatically. Before test discovery and execution it reloads source modules, then
re-imports `gooseapp.conftest`.

See [`running-goose.md`](running-goose.md) for the full reload model.
